# Domain Insight — 领域知识体系化整理

帮助将碎片化的信息源（论文、公众号、媒体解读）整合为持续进化的深度知识文档。

## 知识根目录

用户指定，默认为当前对话上下文中出现的路径。
当前已知根目录：`/Users/mobvoi/Documents/projects/opencode/agent/learning/architecture/`

## 目录结构规范

每个知识主题下：

```
{topic}/
├── README.md          # 主题概览 + 来源索引（元数据）
├── {topic}.md         # 核心知识文档（活文档，持续更新）
└── sources/           # 每个来源的原始摘录笔记
    └── YYYY-MM-DD-{来源名称}.md
```

**与现有结构兼容：** 已存在 `{topic}.md` 的主题，在其同级新建 `sources/` 目录即可，不改动原文件路径。

## 三个核心工作流

### 工作流 A：新建主题

触发：「整理一个新主题」「帮我建一个关于 XX 的知识页」

1. 确认主题名称和所属目录
2. 判断域类型（技术/商业/学术/通用），读取 `references/perspective-framework.md` 选择视角组合
3. 读取根目录下的 `FORMAT.md`（若存在），遵循其格式规范
4. 按模板创建 `{topic}.md` 和 `README.md`
5. 若有初始来源，同步创建 `sources/` 下的来源笔记

### 工作流 B：摄入新来源（最常用）

触发：用户提供链接/文章内容/论文/「我看到一篇...」

1. 提取来源内容的核心信息（若是 URL，让用户粘贴正文或关键内容）
2. 判断归属主题：已有主题 → 更新；无对应主题 → 询问是否新建
3. 读取已有 `{topic}.md`，分析新内容与现有知识的关系：
   - **补充**：填充原本空缺的部分
   - **更新**：新内容更准确/更新，在旧内容旁标注 `> 🔄 已有更新理解，见来源 YYYY-MM-DD`
   - **矛盾**：保留两种观点，注明出处和背景差异
   - **深化**：同一观点有了更好的论据或案例
4. 将摘录笔记写入 `sources/YYYY-MM-DD-{来源名}.md`
5. 更新 `{topic}.md` 对应章节
6. 更新 `README.md` 的来源索引

**旧内容保留原则：** 永远不删除已有内容，只在旁边补充或标注。知识演化的轨迹本身有价值。

### 工作流 C：深化视角

触发：「从 XX 角度分析」「补充面试官视角」「加上未来趋势」

1. 确认目标主题和要深化的视角
2. 读取已有文档，避免重复
3. 基于现有知识 + 用户补充信息，生成该视角的深度解读
4. 写入 `{topic}.md` 对应视角章节

## 文档质量标准

遵循以下优先级（来自 FORMAT.md 精神）：

1. **第一性原理优先**：每个结论都要能回答"为什么"，追问到根本原因
2. **观点而非描述**：不是"XX 技术可以做 YY"，而是"XX 技术的本质矛盾在于..."
3. **具体胜过笼统**：用具体数据、案例、对比，不用"很多"、"通常"
4. **视角要真实**：面试官关心的和 CTO 关心的真的不同，不要把同一段话套不同帽子
