# 市场技能热度参考

> 通用参考框架，适用于 AI / 算法 / 后端 / 数据等技术方向。
> 使用方式：在「市场对标」模式中，结合用户的具体方向和经历一起使用。

---

## AI / 算法工程师方向（2025-2026）

### 🔥 高需求（面试必须能讲）

| 技能方向 | 市场信号 | 深度要求 |
|---------|---------|---------|
| LLM 应用开发与 Agent 系统 | 几乎所有 AI 岗位必须 | 能说清楚架构、踩过的坑、和现有框架的对比 |
| RAG 系统（检索增强生成） | 落地场景最多的技术之一 | 能讲检索策略、rerank、幻觉处理 |
| 模型评测体系 | 工程化 AI 的核心基础设施 | 能讲自动化评测、人工评测、指标设计 |
| 大模型训练 / 微调 | SFT / LoRA / 分布式训练 | 有过实际训练经验的候选人稀缺 |
| Prompt Engineering | 入门门槛低，但深度要求高 | 能讲 CoT / few-shot / 系统化优化方法论 |
| 生产部署与可观测性 | 模型上线后的稳定性保障 | 监控、日志分析、异常检测 |

### 📈 稳定需要（有就加分）

| 技能 | 说明 |
|------|------|
| Python 工程能力 | 不只是会写，要有大规模数据处理经验 |
| 向量数据库 | Pinecone / Qdrant / Milvus 等，RAG 系统标配 |
| 容器化与 CI/CD | Docker / K8s / Jenkins，算法工程师有这个经验加分 |
| 多模态（语音/图像/视频） | 有实际项目经验的候选人少，有竞争力 |
| 数据质量与 Pipeline | 数据清洗、筛选、去重的系统化方法 |

### ⚠️ 常见误区

| 误区 | 真实情况 |
|------|---------|
| "跑过这些 benchmark" | 面试官真正想听的是你对指标局限性的理解，不是分数高低 |
| "用了 LangChain" | 比"用了什么框架"更重要的是"遇到了什么问题、怎么解决的" |
| "模型比之前好了 X%" | 比数字更重要的是：你怎么发现的、为什么好了、是否稳定 |
| "参与了大规模数据处理" | 要能说清楚：数据里有什么噪声、怎么识别、怎么处理 |

## 后端工程师方向（2025-2026）

### 🔥 高需求

| 技能 | 市场信号 |
|------|---------|
| 分布式系统 | 几乎所有中大型公司必问，系统设计面试核心 |
| 高并发 / 高可用架构 | 实际落地经验比理论更重要 |
| 数据库（SQL + NoSQL） | 索引、事务、缓存策略要能讲深 |
| Kubernetes / 云原生 | 生产环境经验加分明显 |
| 性能优化 | 能讲具体的定位方法和优化手段 |

### 📈 稳定需要

Go / Java / Python 工程能力、消息队列（Kafka）、服务网格、可观测性（Prometheus/Grafana）

## 通用原则：如何判断一个技能"值钱"

```
价值 = 稀缺性 × 需求强度 × 你能讲清楚的深度

稀缺性：会这个的人少不少？
需求强度：这个岗位必须要这个技能吗？
讲清楚的深度：你能讲 L4 认知还是只能讲 L1 事实？
```

**最有价值的表达**：
不是"我会 XX"，而是"我用 XX 解决了 YY 问题，遇到了 ZZ 坑，得出了 WW 判断"

## 优势区 / 补强区 / 盲区 分析框架

在做市场对标时，按以下逻辑分类：

**优势区**（你有且市场要）
→ 重点准备，深化到 L4，找到差异化表达

**补强区**（你有但没讲清楚）
→ 升级表达：加决策逻辑、加真实教训、加前瞻判断

**盲区**（市场要但你没有）
→ 两种处理方式：
  1. 如实说边界（"我了解原理，没有实际做过"）
  2. 讲相关经历（"没有直接做过，但在 XX 场景里接触过边缘部分"）
→ 绝对不要伪造，面试官追问两个问题就穿帮
