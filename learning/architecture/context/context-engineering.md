# 上下文工程

> 适用场景：技术面试 / Agent 系统设计 / 长程任务优化

---

## 核心概念（一句话定义）

上下文工程是"在每次 LLM 调用前，如何以可复用、可度量、可演进的方式，拼装并优化输入上下文"——**上下文是有限资源，工程化管理它是 Agent 从"能用"到"好用"的关键跨越**。

---

## 具体解释

### 为什么需要上下文工程

**上下文腐蚀（Context Rot）**：随着 context window 中的 tokens 增加，模型从中准确回忆信息的能力反而下降。这不是偶然，而是 Transformer 架构的内在约束——每个 token 与所有其他 token 建立 $n^2$ 级别的注意力关系，上下文越长，注意力被"拉薄"，对关键信息的聚焦能力下降。

实验验证：大多数主流模型在 256K→1M tokens 的大海捞针测试中，检索命中率从 95%+ 下降到 30-70%。

这意味着：**上下文必须被视作一种有限资源，具有边际收益递减。** 就像人类有有限的工作记忆容量，LLM 有有限的"注意力预算"——每新增一个 token，都在消耗这笔预算。

### 上下文工程 vs 提示工程

| 维度 | 提示工程 | 上下文工程 |
|------|---------|-----------|
| 关注点 | 如何编写指令让模型理解 | 如何管理整个上下文状态 |
| 时间范围 | 单次对话 | 整个 Agent 运行周期 |
| 内容范围 | 提示本身 | 系统提示 + 工具 + 记忆 + 历史 + MCP |
| 核心问题 | 怎么提问 | 哪些信息应该在有限窗口中出现 |

## GSSC 流水线

上下文工程的核心实践是 **GSSC（Gather → Select → Structure → Compress）** 四阶段流水线：

### Gather（汇集）
从多个来源汇集候选信息：
- 系统指令（最高优先级，始终保留）
- 记忆系统检索（相关历史、用户偏好）
- RAG 检索（外部知识库）
- 对话历史（最近 N 条）
- 自定义数据包

### Select（筛选）
基于**相关性 × 新近性**评分，从候选信息中选出最有价值的部分，确保 token 预算不超限。

### Structure（结构化）
将筛选后的信息按优先级组织，确保高价值信息在模型容易"看到"的位置。

### Compress（压缩）
对超长内容做摘要，保留核心信息，丢弃噪声。

**追求的目标**：用尽可能少、但高信号密度的 tokens，最大化获得期望结果的概率。

## 有效上下文的组成

### 系统提示（System Prompt）

常见两极误区：
- **过度硬编码**：塞入复杂的 if-else 逻辑，长期维护成本高、容易出错
- **过于空泛**：只有宏观目标，缺少具体信号

建议：
- 分区组织（background / instructions / output format 等）
- 追求"最小必要信息集"——不是最短，而是能完整勾勒期望行为的最小集合
- System Prompt 控制在 1000 tokens 以内（规则太多说明约束设计有问题，应从工具层或流程层解决）

### 工具（Tools）

- 职责单一、相互低重叠
- 入参描述无歧义
- 工具数量控制：太多导致"读说明书"占满注意力

### Few-shot 示例

- 精选多样且典型的示例
- 好示例胜过千言万语
- 不要罗列所有边界条件

## 长程任务的三大策略

### 压缩整合（Compaction）

当对话接近上下文上限时，对历史进行高保真总结，用摘要重启新的 context window：
- 重点保留：架构决策、未解决缺陷、实现细节
- 丢弃：重复的工具输出、临时的中间过程
- 摘要格式固定（原始目标 + 技术栈 + 完成里程碑 + 关键决策 + 文件状态）
- 摘要不再参与二次压缩（防止层层失真）

### 结构化笔记（Structured Note-taking）

Agent 以固定频率将关键信息写入**上下文外的持久化存储**，在后续按需拉回：
- 维护 TODO 列表、关键发现、阻塞点索引
- 跨上下文重置保持进度一致性
- 每次交互结束追加"Todo Recap"，让模型始终知道"当前在哪一步"

### 子代理架构（Sub-agent Architecture）

主 Agent 负责高层规划与综合，多个子 Agent 在"干净的 context window"中各自深挖：
- 子 Agent 完成探索后只回传 1000-2000 tokens 的凝练摘要
- 主 Agent 避免被庞杂的搜索历史污染
- Boris Cherny（Claude Code 负责人）称为 "context firewall"

**方法选择经验法则**：
- 需要对话连续性 → 压缩整合
- 有里程碑的迭代开发 → 结构化笔记
- 复杂研究 + 并行探索 → 子代理架构

## JIT（Just-in-Time）上下文

从"推理前一次性检索"演进到"即时上下文"：

**传统方式**：预先把所有相关文件加载到 context
**JIT 方式**：维护轻量引用（文件路径、存储查询、URL），运行时通过工具动态加载所需数据

JIT 的优势：
- 上下文始终保持精简
- 允许 Agent 自主导航，实现"渐进式披露"
- 目录层级、命名约定等元数据本身也在传递信号

权衡：运行时探索比预计算检索慢，需要有主见的工程设计确保模型能找到所需工具。

## 分层上下文架构

| 层级 | 内容 | 更新频率 | 作用 |
|------|------|----------|------|
| **L1 系统静态层** | System Prompt + 工具描述 | 几乎不变 | 提供永恒的行为准则和底线 |
| **L2 项目规则层** | CODE_LAW.md / CLAUDE.md | 随项目演进 | 项目特定的规范约束 |
| **L3 动态会话层** | 用户/模型/工具消息历史 | 每轮更新 | 当前任务的状态流转 |

拼接顺序：`L1 → L2 → L3 → 当前输入 → Todo Recap`

分层的意义：模型知道应该优先参考哪一层。L1 的底线规则不会因对话变长而被"稀释"；L2 比 L3 里某条历史消息更权威。

## 工程实践

### 甜蜜区间

实践发现：context window 超过 60% 后性能开始下降。顶级工程师通常将利用率控制在 **60% 以下**。

### 工具输出截断

工具输出是 context 膨胀的最大元凶。推荐方案：截断显示但保留回查路径：
- 截取头尾各 N 行（保留结构信息）
- 完整输出落盘到 `tool-output/` 目录
- 返回 `status: partial` + 完整输出路径
- 模型有主动权决定是否去查完整内容

### @file 注入方式

不要把文件内容直接塞进上下文——用"文件引用 + 工具读取"代替"文件内容预填充"。前者让 Agent 按需访问，后者一次性占满 context 且可能包含大量无关信息。

## 当前发展现状

**技术成熟度**：
- 上下文工程作为独立范式 2025-2026 年才开始被系统讨论，仍处于早期
- GSSC 流水线已在 hello-agents、Claude Code 等框架中有工程实现
- 各框架的 compaction 机制（如 Claude Code 的自动压缩）正在成为标配

**主要矛盾**：
- **上下文长度 vs 注意力质量**：窗口越大不等于越好用，边际效益递减
- **即时检索 vs 预计算**：JIT 更灵活但更慢，需要在任务动态性和时效要求间权衡
- **信息完整 vs 信息精简**：塞得越多越"安全"是错觉，精简才是可靠性的来源

**行业趋势**：
- Context Engineering 取代 Prompt Engineering 成为核心工程能力
- 渐进式披露（Progressive Disclosure）成为大型 Agent 项目的标配
- "注意力预算"意识已成为顶级 Agent 工程师的基本素养

## 未来展望

**短期（1-2年）**：

## 深度理解

**Q: 上下文工程为什么比提示工程更重要？**

> 提示工程解决的是"如何表达单次指令"；上下文工程解决的是"如何管理 Agent 在整个运行周期看到的所有信息"。随着 Agent 任务变长、步骤变多，"哪些信息应该进入 context"的影响远超"某句话怎么措辞"。当任务跨越数十轮循环，提示词已经是次要矛盾。

**Q: 为什么说"把所有信息都塞进去是错的"？**

> 上下文腐蚀——信息量超过某个阈值后，模型注意力被稀释，关键信息反而更难被命中。就像给员工一份 100 页的参考手册，他不可能全部消化，反而容易在查找时出错。精准提供"当前任务最相关的 1% 信息"比提供"所有可能有关的信息"效果更好。

## 一句话总结

上下文工程的本质是在有限注意力预算内最大化信号密度——**不是让 Agent 看到更多，而是让它在对的时机看到对的信息，通过 GSSC 流水线、分层架构和 JIT 检索，把信息流从"大水漫灌"变成"精准滴灌"**。
