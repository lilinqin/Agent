# Agent 本质理解

> 适用场景：技术战略分享、技术选型、团队培训

---

## 什么是 Agent？

很多人把 Agent 简单理解成"会用工具的 LLM"，这个说法没错，但太浅了。

从本质定义切入：**Agent 是一个在环境中持续运作的自主系统——它感知、决策、行动，然后把行动的结果再作为下一次决策的输入。** 这个闭环，才是 Agent 和普通 LLM 调用的根本区别。

我们内部对 Agent 的定位是：**它是第一种能把模糊目标转化为可执行行动序列、并在执行过程中根据反馈动态修正的软件架构。** 这意味着以前只能靠人来完成的任务，现在可以交给 Agent 了。

---

## 架构层面：ReAct 循环

Agent 的运行核心是一个循环：**感知 → 思考 → 行动 → 观察**，不断迭代直到目标达成。

"思考"阶段实际包含两件事：

- **规划**：当前任务怎么拆解
- **工具选择**：下一步用什么手段执行

这两件事如果没有分开意识到，在设计 Agent 系统时就会踩坑。

### Thought / Action / Observation 三段式

```
Thought: 用户想知道北京的天气，需要调用天气查询工具。
Action:  get_weather(city="北京")
Observation: 北京当前天气晴，气温 26 摄氏度。
```

- **Thought**：模型的内部推理快照
- **Action**：结构化的工具调用指令
- **Observation**：执行结果的自然语言摘要，反馈回下一轮

这个设计的精髓在于：**把非结构化的语言推理能力，和结构化的工具调用能力，用一个明确的协议桥接起来。** 这就是 ReAct 范式的本质，也是我们构建 Agent 系统的技术基础。

> 参考论文：[ReAct: Synergizing Reasoning and Acting in Language Models](https://arxiv.org/abs/2210.03629)

## Agent vs Workflow：技术选型的关键决策

| | Workflow | Agent |
|---|---|---|
| 执行方式 | 确定性，按预设流程 | 概率性，自主规划路径 |
| 适用场景 | 规则清晰、边界已知 | 任务模糊、情况多变 |
| 可审计性 | 高 | 低 |
| 灵活性 | 低 | 高 |

我们在技术选型时的判断标准是：**任务的边界是否已知，以及异常情况的处理成本谁更低。**

- 电商退款审批：流程固定、规则清晰 → Workflow 更可靠、更可审计
- 合同风险分析：情况千变万化，分支写不完 → Agent 的主场

**我们认为最优解是混合架构**：用 Workflow 管控主干流程保证稳定性，在需要动态判断的节点嵌入 Agent 提供灵活性。LangGraph 的核心设计思路正是如此——用状态图建模执行流程，让确定性和自主性各司其职。

## Agent 的局限：理性认识技术边界

Agent 的能力上限，本质上取决于三件事：

1. **LLM 的推理质量**
2. **工具集的覆盖范围**
3. **上下文窗口的管理策略**

### 上下文管理是最容易被忽视的工程难题

Agent 每一轮循环都在往 context 里塞东西——Thought、Action、Observation——跑几轮之后 token 就爆了，或者早期信息被稀释掉导致**决策漂移**。

这就是为什么做复杂 Agent 必须认真设计记忆系统，把以下三部分分开管理：

- **短期工作记忆**：当前任务的执行上下文
- **长期知识存储**：跨任务的持久化知识（通常用向量库）
- **工具状态**：工具调用的中间结果缓存

### Agent 越自主，越难调试

确定性系统出错可以追溯每一步，Agent 出错面对的是一个概率采样的黑盒。

我们成熟的 Agent 系统都需要：
- 完善的 **tracing**（记录每一步的 Thought/Action/Observation）
- **人工介入机制**（Human-in-the-loop）

**完全自主是理想，但"可控的自主"才是工程实践。**

## 技术战略思考

我们在 Agent 方向的投入，核心逻辑是什么？

**第一，LLM 是能力基础。** 它提供了语言理解和推理能力，但没有行动能力。

**第二，工具是能力延伸。** 只有接入真实的外部世界——搜索、数据库、API——LLM 才能产生实际价值。

**第三，循环反馈是核心差异。** 传统自动化是"一次性执行"，Agent 是"持续迭代"，这让它能处理以前无法自动化的"模糊任务"。

但我们也要清醒认识到：Agent 带来了新的复杂性——调试成本、不确定性、安全风险。这不是"更聪明的自动化"，而是一种**新的软件范式**，需要新的工程实践来支撑。

## 一句话总结

Agent 不是更聪明的搜索框，也不是自动化脚本的升级版。它是第一种**能把模糊目标转化为可执行行动序列、并在执行过程中根据反馈动态修正**的软件架构。这个特性让它有机会处理以前只能靠人来完成的任务——但代价是必须认真对待它的不确定性。
