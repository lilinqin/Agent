# Agent 本质理解

> 适用场景：技术面试 / 技术战略分享 / 团队培训

---

## 什么是 Agent？

很多人把 Agent 简单理解成"会用工具的 LLM"，这个说法没错，但太浅了。

从本质定义切入：**Agent 是一个在环境中持续运作的自主系统——它感知、决策、行动，然后把行动的结果再作为下一次决策的输入。** 这个闭环，才是 Agent 和普通 LLM 调用的根本区别。

我们内部对 Agent 的定位是：**它是第一种能把模糊目标转化为可执行行动序列、并在执行过程中根据反馈动态修正的软件架构。**

---

## 架构层面：ReAct 循环

Agent 的运行核心是一个循环：**感知 → 思考 → 行动 → 观察**，不断迭代直到目标达成。

"思考"阶段实际包含两件事：

- **规划**：当前任务怎么拆解
- **工具选择**：下一步用什么手段执行

### Thought / Action / Observation 三段式

```
Thought: 用户想知道北京的天气，需要调用天气查询工具。
Action:  get_weather(city="北京")
Observation: 北京当前天气晴，气温 26 摄氏度。
```

这个设计的精髓在于：**把非结构化的语言推理能力，和结构化的工具调用能力，用一个明确的协议桥接起来。** 这就是 ReAct 范式的本质。

> 参考论文：[ReAct: Synergizing Reasoning and Acting](https://arxiv.org/abs/2210.03629)

## Agent vs Workflow

| | Workflow | Agent |
|---|---|---|
| 执行方式 | 确定性，按预设流程 | 概率性，自主规划路径 |
| 适用场景 | 规则清晰、边界已知 | 任务模糊、情况多变 |
| 可审计性 | 高 | 低 |
| 灵活性 | 低 | 高 |

判断标准是：**任务的边界是否已知，以及异常情况的处理成本谁更低。**

我们认为最优解是**混合架构**：用 Workflow 管控主干流程保证稳定性，在需要动态判断的节点嵌入 Agent。LangGraph 的核心设计思路正是如此。

## Agent 的局限

Agent 的能力上限，本质上取决于三件事：

1. **LLM 的推理质量**
2. **工具集的覆盖范围**
3. **上下文窗口的管理策略**

### 上下文管理是最容易被忽视的工程难题

Agent 每一轮循环都在往 context 里塞东西，跑几轮后 token 就爆了，或者早期信息被稀释导致**决策漂移**。

成熟系统都需要：
- 完善的 **tracing**
- **人工介入机制**（Human-in-the-loop）

**完全自主是理想，"可控的自主"才是工程实践。**

## 当前发展现状

**技术成熟度**：Agent 技术从 2022 年 ReAct 论文发表至今约 2 年，已经从"概念验证"进入"产品化"阶段。

**主要矛盾**：
- **可靠性问题**：LLM 的概率性导致 Agent 行为不可预测
- **调试困难**：没有堆栈，只有对话记录
- **成本控制**：多轮调用导致成本快速膨胀

**行业趋势**：
- 所有云厂商都在推 Agent 平台（OpenAI Agents、Claude Agent、腾讯元器等）
- 框架层竞争激烈（LangGraph、AutoGen、AgentScope）
- 工具生态成为竞争焦点（MCP 协议）

---

## 未来展望

**短期（1-2年）**：
- Agent 会从"玩具"变成"工具"，在特定场景实现产品化
- 评测体系会逐步建立（类似 LLM 的 benchmark）
- MCP 等协议会推动工具标准化

**长期（3-5年）**：
- Agent 可能成为软件的"默认形态"——从人找功能变成功能找人
- 多 Agent 协作会诞生"虚拟团队"
- 自主性和可控性的平衡会成为核心挑战

**我们的布局**：
- 核心能力：记忆系统 + 工具平台
- 产品策略：先在垂直场景验证，再扩展
- 技术储备：关注 Agent 评测、多 Agent 协作

## 面试常见问题

**Q: Agent 和普通 LLM 调用的区别？**

> 普通 LLM 是一次性调用，Agent 是循环迭代。核心差异是有反馈机制——Action 的结果（Observation）会反馈给 LLM 做下一轮决策。

**Q: 什么时候用 Workflow 不用 Agent？**

> 规则固定、边界清晰、可审计性要求高的场景用 Workflow（如退款审批）。Agent 适合情况多变、分支写不完的场景（如合同风险分析）。

**Q: Agent 的能力上限取决于什么？**

> 三件事：LLM 本身的推理质量、工具能覆盖多少场景、以及如何管理上下文窗口防止信息稀释。

## 一句话总结

Agent 是第一种**能把模糊目标转化为可执行行动序列、并在执行过程中根据反馈动态修正**的软件架构。这个特性让它有机会处理以前只能靠人来完成的任务——但代价是必须认真对待它的不确定性。
