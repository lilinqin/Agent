# 工具系统

> 适用场景：技术面试 / 工具平台建设 / 技术选型

---

## 工具的本质

**工具是 Agent 与外部世界交互的桥梁。** LLM 本身的知识是静态的、训练截止的，但现实世界是动态的——天气会变、股票会涨、数据库会更新。工具让 Agent 能够获取实时信息，执行具体操作。

我们把工具定位为：**Agent 能力的延伸。** LLM 负责"脑"，工具负责"手"。

---

## 工具定义三要素

```json
{
  "name": "search",
  "description": "当你需要查询实时信息时使用",  // 触发场景
  "parameters": { ... }
}
```

精髓在于：**工具描述的重点是"什么时候用"，而不是"怎么用"。**

我们踩过的坑：之前有工程师写"调用 SerpApi，传入 query 参数返回搜索结果"——这是给开发者看的，不是给 LLM 看的。

---

## Function Calling vs. ReAct

### Function Calling

模型直接输出结构化参数：
```json
{"name": "search", "arguments": {"query": "北京天气"}}
```
- **优点**：格式可靠，无需解析
- **缺点**：需要模型支持

### ReAct 格式

模型输出自然语言：
```
Action: search[北京天气]
```
- **优点**：通用性强
- **缺点**：需要正则解析

**我们目前的实践**：生产环境优先选 Function Calling。

> 参考论文：[Toolformer](https://arxiv.org/abs/2302.04761)

## 工具类型

| 类型 | 示例 | 场景 |
|------|------|------|
| **查询型** | 搜索、数据库查询 | 获取外部信息 |
| **执行型** | 发送邮件、执行代码 | 执行具体操作 |
| **计算型** | 计算器、公式解析 | 精确计算 |
| **检索型** | 向量搜索、知识库查询 | RAG 场景 |

## 工程实践

### 描述即提示词

```python
# ❌ 错误
"调用 SerpApi，传入 query 参数"

# ✅ 正确
"当你需要查询实时信息时使用"
```

### 错误处理

临时错误重试，永久错误返回明确信息让 LLM 调整。

### 重试机制

- 临时错误：自动重试 2-3 次
- 永久错误：返回明确错误信息

## 面试常见问题

**Q: 工具描述怎么写最有效？**

> 描述"什么时候用"而非"怎么用"。例如"当你需要查询实时天气时使用"而不是"调用天气API"。

**Q: ReAct 和 Function Calling 怎么选？**

> Function Calling 格式可靠但需要模型支持；ReAct 更通用但需要解析。生产环境首选 Function Calling。

**Q: 工具调用失败怎么办？**

> 分情况处理：临时错误（网络波动）重试；参数错误返回具体错误让 LLM 调整；权限错误触发人工介入。

**Q: 如何防止工具调用滥用？**

> 1. 限制调用次数；2. 记录历史避免重复；3. 敏感操作人工确认；4. 工具级别权限控制。
