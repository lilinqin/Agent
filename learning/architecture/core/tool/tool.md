# 工具系统

> 适用场景：技术面试 / 工具平台建设 / 技术选型

---

## 核心概念（一句话定义）

工具是 Agent 与外部世界交互的桥梁——**LLM 负责"脑"（推理规划），工具负责"手"（获取实时信息、执行具体操作）**，二者的分工决定了 Agent 的能力边界。

---

## 工具定义三要素

```json
{
  "name": "search",
  "description": "当你需要查询实时信息时使用",
  "parameters": { ... }
}
```

精髓在于：**工具描述的重点是"什么时候用"，而不是"怎么用"。**

我们踩过的坑：之前有工程师写"调用 SerpApi，传入 query 参数返回搜索结果"——这是给开发者看的，不是给 LLM 看的。LLM 需要的是场景触发信号，而不是 API 文档。

---

## 工具调用方式

### Function Calling

模型直接输出结构化参数：
```json
{"name": "search", "arguments": {"query": "北京天气"}}
```
- **优点**：格式可靠，无需正则解析
- **缺点**：需要模型原生支持

### ReAct 格式

模型输出自然语言：
```
Action: search[北京天气]
```
- **优点**：通用性强
- **缺点**：需要正则解析，脆弱性高

**生产实践**：优先选 Function Calling。ReAct 格式用于不支持 Function Calling 的模型。

## 工具设计的 Goldilocks 原则

工具既不是越多越好，也不是越原子越好——**万能工具**（一个工具包打天下）和**过度原子化**（每个细节都是独立工具）都会失败：

- **万能工具的问题**：多步操作在一个 Action 内，中间状态丢失，出错不可诊断，模型只看到最终结果无法针对性纠错
- **过度原子化的问题**：工具数量爆炸，Schema 占满上下文，模型陷入"选工具困难症"，决策负担反超执行本身

### 三层设计架构（推荐）

| 层级 | 代表工具 | 设计目标 | 典型约束 |
|------|---------|---------|---------|
| **高频原子层** | LS / Glob / Grep / Read | 一步一证据，稳定可诊断 | 强类型输入输出，明确错误码 |
| **中频受控层** | Write / Edit / MultiEdit | 改动可验证、可回滚 | 强制"先读后改"，乐观锁 |
| **低频兜底层** | Bash | 处理原子工具覆不到的边角需求 | 明确禁区，不走主链路 |

**判断框架**：高频 + 强确定 → 必须原子化；中频 + 有副作用 → 必须受控；低频 + 弱确定 → 保留弹性但明确禁止什么。

### 可诊断性原则

**可诊断性是可恢复性的前提。** 如果不知道哪坏了，就修不好。

统一响应协议让所有工具说同一种语言：
```json
{
  "status": "success" | "partial" | "error",
  "data": { ... },
  "text": "人类可读摘要",
  "error": { "code": "NOT_FOUND", "message": "具体错误" }
}
```

模型能清晰区分"真的没找到"（`status: success, data: []`）和"执行出错"（`status: error, error.code: NOT_FOUND`）。

## 通信协议标准化（MCP / A2A / ANP）

### MCP（Model Context Protocol）—— Anthropic 提出

**解决问题**：标准化 Agent 与外部工具/资源的通信方式，不再需要为每个服务单独写适配器。

设计哲学是"上下文共享"——不只是 RPC 调用，还允许共享代码结构、依赖关系、提交历史等丰富上下文，让 Agent 做出更智能的决策。

**适用场景**：Agent 需要访问文件系统、数据库、GitHub、Slack 等外部服务。

### A2A（Agent-to-Agent Protocol）—— Google 提出

**解决问题**：实现 Agent 之间的点对点通信，让 Agent 像人类团队一样进行对话、协商和协作。

设计哲学是"对等通信"——每个 Agent 既是服务提供者，也是服务消费者，避免中心化协调器的瓶颈。

**适用场景**：多个专业 Agent 协作完成任务（研究员 + 撰写员 + 编辑）。

### ANP（Agent Network Protocol）—— 开源社区

**解决问题**：大规模 Agent 网络中的服务发现和路由，让 Agent 动态找到所需服务，而不需要预先配置所有连接关系。

**适用场景**：构建包含成百上千个 Agent 的大规模生态系统。

### 协议选择指南

| 场景 | 推荐协议 |
|------|---------|
| 访问外部服务（文件、数据库、API） | MCP |
| 多 Agent 协作完成任务 | A2A |
| 大规模 Agent 生态系统 | ANP |

## 工程实践

### 工具数量控制

> "工具越少而精越好" —— Vercel 工程团队

工具过多导致决策瘫痪。Claude Code 约 20 个工具，团队仍在审视是否全部必要。Vercel 将完整工具库精简到核心工具后，速度和可靠性显著提升。

**经验法则**：如果人类工程师都说不准用哪个工具，别指望 Agent 做得更好。

### 描述即提示词

```python
# 错误写法（给开发者看的）
"调用 SerpApi，传入 query 参数"

# 正确写法（给 LLM 看的）
"当你需要查询实时信息时使用"
```

### 错误处理

- **临时错误**：自动重试 2-3 次
- **永久错误**：返回明确错误信息，让 LLM 调整策略
- **熔断机制**：工具连续失败（3次）临时禁用，防止死循环

### "先读后改"约束

对于 Write/Edit 类操作，强制先 Read 才能改：
- 防止模型"凭记忆"修改文件——必须先把文件内容拿到上下文中确认
- 配合乐观锁：如果文件在 Read 之后被外部修改，Edit 返回 `CONFLICT` 错误

## 当前发展现状

**技术成熟度**：
- Function Calling 已是事实标准，所有主流模型支持
- MCP 协议快速成为工具标准化的主流路径（Anthropic 力推）
- 三层工具架构逐渐成为 Code Agent 的工程最佳实践

**主要矛盾**：
- **碎片化 vs 标准化**：MCP 还处于生态建设期，工具质量参差不齐
- **工具数量 vs 能力覆盖**：工具越精简可靠性越好，但覆盖需求越窄
- **可诊断性 vs 灵活性**：原子工具步骤繁琐，但出错必须可定位才能可恢复

**行业趋势**：
- Harness Engineering 视角：工具少而精是硬原则，不是软建议
- 模型内化工具能力（programmatic tool use、tool search）正在发生
- 工具的执行轨迹本身就是高价值训练数据

## 深度理解

**Q: 工具描述怎么写最有效？**

> 描述"什么时候用"而非"怎么用"——"当你需要查询实时信息时使用"而不是"调用天气 API"。LLM 需要的是场景判断信号，而不是 API 文档。描述越像"触发条件"，模型越能准确选择工具。

**Q: 工具越多越好吗？为什么反而要减少工具？**

> 工具多会导致两个问题：1）Schema 占据大量上下文，稀释有限注意力；2）工具功能重叠让 Agent 陷入"选哪个"的决策困境。Claude Code 团队的实践验证：精简到核心工具集后，Agent 可靠性显著提升。工具箱的价值不在于"扳手数量"，而在于"几把万能扳手如何高效组合"。

**Q: 为什么要区分"高频原子层"和"低频兜底层"？**

> 两类工具的失败模式不同。高频工具（Glob/Grep/Read）每天调用几十次，一旦出错必须立刻可诊断；低频工具（Bash）调用频率低，接受一定灵活性但必须设置禁区。如果 Bash 能做 Grep 的事，模型就会用 Bash——管道命令把错误吞掉，可诊断性消失，调试变成噩梦。
