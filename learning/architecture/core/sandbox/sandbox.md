# 安全沙箱

> 适用场景：技术面试 / 安全架构设计 / 生产系统规划

---

## 为什么需要沙箱

Agent 的核心特性是"自主行动"——它可以调用工具、执行代码、访问数据。自主性意味着风险：

- 执行恶意代码
- 过度调用 API 导致经济损失
- 泄露敏感数据
- 行为越界无法恢复

我们把安全视为**底线**，而不是锦上添花。沙箱的本质目标是：**在保证 Agent 能力的前提下，限制其破坏范围。**

---

## 沙箱多层防御

### 权限控制

```python
class ToolPermissions:
    allowed_tools = {"search", "calculator", "fetch_data"}
    denied_tools = {"exec", "delete_db", "send_email"}
```

精髓在于：**用白名单而非黑名单**——默认拒绝，只开放明确需要的工具。

### 资源限制

| 限制项 | 说明 | 示例 |
|--------|------|------|
| **调用频率** | 单位时间最大调用次数 | 每分钟最多 10 次 |
| **调用总量** | 单任务最大调用次数 | 最多 50 次 |
| **成本配额** | API 成本限制 | 每次任务最多 $1 |

### 执行隔离

代码执行是最大风险点：
- 绝不能直接在宿主环境执行用户代码
- 使用容器化（Docker）或沙箱环境

```python
# ❌ 危险
eval(code)

# ✅ 安全
container = create_sandbox_container()
result = container.run(code, timeout=5)
```

### 输出过滤

敏感信息脱敏（信用卡号、API Key 等）。

## Heartbeat：心跳机制

Agent 的"健康检查"机制。

### 超时控制

```python
async def run_with_heartbeat(agent, task, timeout=300):
    start = time.time()
    while not agent.is_done():
        if time.time() - start > timeout:
            agent.stop()
            return "任务超时"
```

### 人工介入（Human-in-the-loop）

关键节点暂停等待确认：
- 敏感操作（删除、支付）
- 高风险决策（法律、医疗）
- Agent 置信度低于阈值时

## 当前发展现状

**技术成熟度**：
- 安全实践逐步标准化
- 主流框架内置安全机制
- 但仍以"打补丁"为主

**主要矛盾**：
- **灵活性 vs 安全性**：越安全限制越多，Agent 能力受限
- **成本 vs 安全**：完善的安全机制成本高
- **检测 vs 防御**：很多问题难以提前检测

**行业趋势**：
- Agent 安全成为热点
- MCP 协议在推动安全标准化
- 出现专门的 Agent 安全公司

## 面试常见问题

**Q: Agent 安全要考虑哪些方面？**

> 1. 工具权限控制——白名单而非黑名单；2. 资源限制——频率、配额、成本；3. 执行隔离——代码必须在沙箱运行；4. 输出过滤——敏感信息脱敏；5. 心跳机制——超时控制和人工介入。

**Q: 怎么防止 Agent 调用过度？**

> 三层防护：1. 单轮 max_steps；2. 单任务总调用次数；3. 单位时间频率限制。加上成本监控。

**Q: Human-in-the-loop 适用场景？**

> 1. 敏感操作（删除、支付）；2. 高风险决策（法律、医疗）；3. 不确定情况——当 Agent 置信度低于阈值时触发。

**Q: 代码执行为什么需要沙箱？**

> Agent 可能被诱导执行恶意代码。必须隔离：1. 容器化运行；2. 超时限制；3. 资源配额；4. 网络隔离。

---

## 一句话总结

沙箱本质是**在保证能力的前提下限制破坏范围**：权限控制定边界、资源限制防过度、执行隔离保安全、心跳机制兜底。**完全自主是理想，"可控的自主"才是工程现实。**
