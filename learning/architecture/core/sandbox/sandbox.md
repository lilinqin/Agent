# 安全沙箱

> 适用：技术面试、安全架构

---

## 为什么需要沙箱

Agent 的核心是**自主行动**——调用工具、执行代码、访问数据。但自主性意味着风险：

- 执行恶意代码
- 过度调用 API 导致经济损失
- 泄露敏感数据
- 行为越界无法恢复

**沙箱目标**：保证能力的前提下，限制破坏范围。

---

## 多层防御

### 1. 权限控制
白名单机制——只有明确允许的工具才能调用。

### 2. 资源限制
- **调用频率**：每分钟最多 N 次
- **调用总量**：单任务最多 M 次
- **成本配额**：每次任务最多花费 $X

### 3. 执行隔离
- 代码必须在沙箱/容器中运行，绝不直接 eval()
- 限制网络访问，禁止内网

### 4. Guardrails
- **输入过滤**：恶意指令、敏感词
- **输出过滤**：敏感信息脱敏、有害内容检测

---

## Heartbeat 机制

Agent 的"健康检查"：

1. **超时控制**：任务运行超时自动终止
2. **人工介入**：关键节点暂停等待确认
   - 敏感操作（删除、支付）
   - 高风险决策（法律、医疗）
   - Agent 置信度低于阈值时

---

## 面试高频

**Q: Agent 安全要考虑哪些方面？**
> "1. 工具权限控制——白名单而非黑名单；2. 资源限制——频率、配额、成本；3. 执行隔离——代码必须在沙箱运行；4. 输出过滤——敏感信息脱敏；5. 心跳机制——超时控制和人工介入。"

**Q: 怎么防止 Agent 调用过度？**
> "三层防护：1. 单轮 max_steps；2. 单任务总调用次数；3. 单位时间频率限制。加上成本监控。"
