# 记忆系统

> 适用场景：技术面试 / 长对话系统设计 / RAG 规划

---

## 为什么需要记忆

LLM 有上下文窗口限制——GPT-4 最大 128K tokens，但仍有限制。

**Agent 运行多轮后面临两个致命问题：**

1. **上下文膨胀**：每轮循环都加 Thought/Action/Observation，几轮后 token 就满了
2. **信息稀释**：早期关键信息被淹没，LLM "忘记"了

这就是为什么做复杂 Agent 必须认真设计记忆系统。

---

## 记忆分层架构

| 层次 | 存储 | 生命周期 | 作用 |
|------|------|----------|------|
| **工作记忆** | context | 单轮任务 | 当前任务执行上下文 |
| **会话记忆** | 变量/DB | 会话期间 | 多轮对话历史 |
| **长期记忆** | 向量数据库 | 持久化 | 跨会话知识 |

### 工作记忆

Agent 当前任务的执行上下文。精髓在于：每次 run() 开始时初始化，超过 max_steps 自动截断。

### 会话记忆

多轮对话的历史，本质是消息列表，需要长度控制。

### 长期记忆

跨会话的持久化知识存储，用向量数据库。

精髓在于：不是一股脑塞入 context，而是用向量检索"召回"相关内容。

> 参考论文：[MemGPT](https://arxiv.org/abs/2310.08560)

---

## 记忆管理策略

### 截断

只保留最近 N 轮或 M tokens。简单但可能丢失重要信息。

### 摘要

每轮结束后用 LLM 生成摘要：
```python
def summarize(messages):
    prompt = f"将以下对话压缩成简短摘要，保留关键信息：\n{messages}"
    return llm.invoke(prompt)
```
- **优点**：压缩率高
- **缺点**：每次都要调用 LLM，有成本

### RAG（检索增强）

把历史当知识库，用向量检索"召回"相关内容。

适用：长程任务、需要参考历史经验。

## 当前发展现状

**技术成熟度**：
- 分层架构已成为共识
- RAG 技术成熟
- 摘要技术仍在发展中

**主要矛盾**：
- **成本 vs 效果**：摘要成本高，但效果好
- **检索质量**：向量检索效果依赖 embedding 质量
- **一致性**：多层次记忆的一致性维护困难

**行业趋势**：
- MemGPT 带火了层级记忆概念
- 上下文窗口快速扩展（Claude 3 达到 200K）
- 记忆压缩技术受到关注

## 面试常见问题

**Q: 上下文窗口管理有哪些策略？**

> 1. 截断——只保留最近 N 轮；2. 摘要——用 LLM 压缩历史；3. RAG——向量检索召回相关历史。根据任务复杂度选择。

**Q: Agent 记忆和普通对话历史有什么区别？**

> 普通对话只需存消息列表。Agent 记忆要区分工作记忆（当前任务上下文）、会话记忆（多轮历史）、长期记忆（跨会话知识），每层管理策略不同。

**Q: 记忆系统如何影响 Agent 性能？**

> 好的记忆设计让 Agent 不丢上下文；差的设计导致信息稀释，造成"决策漂移"——Agent 忘了最初目标，越跑越偏。

**Q: RAG 做记忆和普通 RAG 有什么区别？**

> 本质一样，都是向量检索。区别是检索内容：普通 RAG 检索外部知识库，记忆 RAG 检索历史对话。

## 一句话总结

记忆系统核心是**分层管理**：工作记忆保执行、会话记忆保连贯、长期记忆保知识。三层各司其职，通过截断/摘要/检索策略控制上下文长度。
