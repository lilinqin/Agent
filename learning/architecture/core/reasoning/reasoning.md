# 智能体推理范式

> 适用场景：技术面试 / 技术架构设计 / 技术选型

---

## 核心区别

| 范式 | 本质 | 适用场景 |
|------|------|----------|
| **ReAct** | 动态决策，每步根据上一步结果调整 | 需要外部工具、实时信息、开放式任务 |
| **Plan-and-Solve** | 先规划再执行，不中途调整 | 结构化多步骤任务、逻辑确定 |
| **Reflection** | 事后检查，迭代优化 | 高准确性要求 |

---

## ReAct

**核心循环**：
```
Thought: 分析当前情况
Action: 调用工具
Observation: 结果反馈
→ 下一个 Thought...
```

精髓在于：**LLM 负责"运筹"，工具负责"执行"，分工明确。**

### 为什么有效

- Observation 反馈机制实现**动态纠错**——搜索结果不行，立刻换搜索词
- 不需要预先规划完整路径，走一步看一步

### 适用场景

- 实时信息查询（天气、新闻、股价）
- 调用外部 API（查数据库、第三方服务）

### 局限

- 强依赖 LLM 本身能力
- 每步调用，成本高
- 可能陷入局部最优

> 参考论文：[ReAct: Synergizing Reasoning and Acting](https://arxiv.org/abs/2210.03629)

---

## Plan-and-Solve

**两阶段**：规划 → 执行

精髓在于把**规划**和**执行**解耦。规划阶段专注拆解任务，执行阶段专注单步求解。

### 适用场景

- 多步数学应用题
- 代码生成（先设计结构再写代码）
- 报告撰写（先列大纲再填充）

### 局限

- 规划错则全错
- 中途无法改计划

---

## Reflection

**三步循环**：执行 → 反思 → 优化 → 重复

本质是让 LLM 写完初稿后检查一遍，再改。用额外调用换准确率。

### 适用场景

- 合同审查、数据分析（不容出错）
- 代码优化

### 代价

调用次数翻倍以上。

---

## 工程实践

### 工具定义

```json
{
  "name": "search",
  "description": "当你需要查询实时信息时使用"
}
```

**精髓**：告诉 LLM 什么时候用，不是怎么用。

### 安全措施

- **max_steps**：防止死循环
- **正则解析**：提取 Thought/Action
- **异常处理**：工具调用失败

## 当前发展现状

**技术成熟度**：
- ReAct 是最成熟的范式，论文引用量最高
- Plan-and-Solve 在数学、代码场景效果好
- Reflection 在"不容有错"场景是事实标准

**主要矛盾**：
- **成本 vs 效果**：ReAct 每步都调用 LLM，成本高；Reflection 更准确但成本更高
- **灵活 vs 稳定**：ReAct 灵活但不可预测，Plan-and-Solve 稳定但不灵活
- **规划 vs 涌现**：是预先规划还是让 LLM 自由涌现

**行业趋势**：
- 各大框架都在优化 Agent 循环效率
- 出现了"Plan-and-Execute"（Plan-and-Solve 的改进版）
- 多范式组合成为趋势

---

## 未来展望

**短期（1-2年）**：
- 范式组合会成为主流（Plan + ReAct + Reflection）
- 成本优化是关键点（缓存、蒸馏）
- 评测会逐步标准化

**长期（3-5年）**：
- 可能出现新的"涌现"范式
- LLM 本身能力提升会改变范式选择
- "自适应推理"可能取代固定范式

**我们的布局**：
- 核心：ReAct 为基础，叠加 Reflection 提升准确率
- 优化：实现智能路由，根据任务类型选择范式
- 储备：关注新型范式发展

## 面试常见问题

**Q: 三种范式核心区别？**

> ReAct 动态决策灵活但成本高；Plan-and-Solve 稳定但缺乏调整；Reflection 用额外调用换准确率。

**Q: 什么时候用 ReAct？**

> 需要外部工具、实时信息的任务。LLM 知识库没有实时数据，必须靠工具。

**Q: Agent 陷入循环怎么办？**

> 1. 加 max_steps 限制；2. 记录历史避免重复调用；3. 用 Reflection 做兜底检查。

**Q: 工具描述怎么写有效？**

> "当你需要查询实时信息时使用此工具"，而不是 "调用 SerpApi 传入 query 参数"。

## 一句话总结

ReAct 解决动态决策问题，Plan-and-Solve 解决稳定性问题，Reflection 解决准确率问题。**三种范式不互斥，实际系统往往是组合使用。**
