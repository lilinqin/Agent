# 智能体推理范式

> 适用场景：技术架构设计、技术选型、团队培训

---

## 核心区别

我们在构建 Agent 系统时，首先要回答一个问题：**Agent 应该如何思考？**

| 范式 | 本质 | 适用场景 |
|------|------|----------|
| **ReAct** | 动态决策，每步根据上一步结果调整 | 需要外部工具、实时信息、开放式任务 |
| **Plan-and-Solve** | 先规划再执行，不中途调整 | 结构化多步骤任务、逻辑确定 |
| **Reflection** | 事后检查，迭代优化 | 高准确性要求 |

---

## ReAct

**核心循环**：
```
Thought: 分析当前情况
Action: 调用工具
Observation: 结果反馈
→ 下一个 Thought...
```

很多人觉得 ReAct 复杂，其实它的本质很朴素：**LLM 负责"运筹"，工具负责"执行"，分工明确。**

### 这个设计解决的核心问题

我们为什么要用"思考-行动-观察"这种看似繁琐的循环？

**因为现实世界是动态的。** 你无法在开始前规划完整路径——每一步的行动都可能带来新的信息，需要根据反馈调整下一步。

### 为什么有效

- Observation 反馈机制实现了**动态纠错**——搜索结果不行，立刻换搜索词
- 不需要预先规划完整路径，走一步看一步

### 适用场景

- 实时信息查询（天气、新闻、股价）
- 需要精确计算（数学题、统计）
- 调用外部 API（查数据库、第三方服务）

### 局限

- 强依赖 LLM 本身能力（格式 follow、推理能力）
- 每步调用，复杂任务成本高
- 可能陷入局部最优

> 参考论文：[ReAct: Synergizing Reasoning and Acting](https://arxiv.org/abs/2210.03629)

---

## Plan-and-Solve

**两阶段**：
1. 规划：LLM 分解任务，输出步骤列表
2. 执行：按顺序执行，每步用上一步结果

这个设计的精髓在于把**规划**和**执行**解耦。规划阶段专注拆解任务，执行阶段专注单步求解。

### 适用场景

我们会在以下场景选择 Plan-and-Solve：
- 多步数学应用题
- 代码生成（先设计结构再写代码）
- 报告撰写（先列大纲再填充）

### 局限

- 规划错则全错
- 中途无法改计划
- 任务模糊时无法规划

---

## Reflection

**三步循环**：执行 → 反思（LLM 挑毛病）→ 优化 → 重复

本质就是让 LLM 写完初稿后检查一遍，再改。多跑几轮直到检查不出错。

### 适用场景

我们会在"不容有错"的场景使用：
- 合同审查、数据分析
- 复杂推理（数学证明、逻辑推导）
- 代码优化（初版 → 反思性能 → 优化）

### 代价

调用次数翻倍以上，但**用额外调用换准确率**。

## 工程实践

### 工具定义

```json
{
  "name": "search",
  "description": "当你需要查询实时信息时使用"
}
```

**精髓**：告诉 LLM 什么时候用，不是怎么用。

### 安全措施

- **max_steps**：防止死循环
- **正则解析**：提取 Thought/Action
- **异常处理**：工具调用失败

---

## 技术选型思考

我们在选择推理范式时，核心考量是什么？

**第一，任务性质。** 开放式任务用 ReAct，结构化任务用 Plan-and-Solve。

**第二，准确率要求。** 高要求场景加 Reflection 迭代。

**第三，成本控制。** ReAct 每步都调用 LLM，成本高；Plan-and-Solve 只需规划时调用，执行复用。

## 一句话总结

ReAct 解决动态决策问题，Plan-and-Solve 解决稳定性问题，Reflection 解决准确率问题。**三种范式不互斥，实际系统往往是组合使用。**
