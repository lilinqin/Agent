# 智能体推理范式

> 适用场景：技术面试、系统设计

---

## 核心区别

很多人问：ReAct、Plan-and-Solve、Reflection 有什么区别？该用哪个？

| 范式 | 本质 | 适用场景 |
|------|------|----------|
| **ReAct** | 动态决策，每步根据上一步结果调整 | 需要外部工具、实时信息、开放式任务 |
| **Plan-and-Solve** | 先规划再执行，不中途调整 | 结构化多步骤任务、逻辑确定 |
| **Reflection** | 事后检查，迭代优化 | 高准确性要求 |

---

## ReAct

**核心循环**：
```
Thought: 分析当前情况
Action: 调用工具
Observation: 结果反馈
→ 下一个 Thought...
```

很多人觉得 ReAct 复杂，其实它的本质很朴素：**LLM 负责"运筹"，工具负责"执行"，分工明确。**

**为什么有效**：
- Observation 反馈机制实现了**动态纠错**——搜索结果不行，立刻换搜索词
- 不需要预先规划完整路径，走一步看一步

**适用场景**：
- 实时信息查询（天气、新闻、股价）
- 需要精确计算（数学题、统计）
- 调用外部 API（查数据库、第三方服务）

**局限**：
- 强依赖 LLM 本身能力（格式 follow、推理能力）
- 每步调用，复杂任务成本高
- 可能陷入局部最优

> 参考论文：[ReAct: Synergizing Reasoning and Acting](https://arxiv.org/abs/2210.03629)

---

## Plan-and-Solve

**两阶段**：
1. 规划：LLM 分解任务，输出步骤列表
2. 执行：按顺序执行，每步用上一步结果

这个设计的精髓在于把**规划**和**执行**解耦。规划阶段专注拆解任务，执行阶段专注单步求解。

**适用场景**：
- 多步数学应用题
- 代码生成（先设计结构再写代码）
- 报告撰写（先列大纲再填充）

**局限**：
- 规划错则全错
- 中途无法改计划
- 任务模糊时无法规划

---

## Reflection

**三步循环**：执行 → 反思（LLM 挑毛病）→ 优化 → 重复

本质就是让 LLM 写完初稿后检查一遍，再改。多跑几轮直到检查不出错。

**代价**：调用次数翻倍以上，但用额外调用换准确率。

**适用场景**：
- 合同审查、数据分析（不容出错）
- 复杂推理（数学证明、逻辑推导）
- 代码优化（初版 → 反思性能 → 优化）

## 工程要点

### 工具定义

```json
{
  "name": "search",
  "description": "当你需要查询实时信息时使用"  // 告诉 LLM 什么时候用，不是怎么用
}
```

### ReAct 提示词格式

```
Thought: ...
Action: Search[query]
```

### 安全措施

- **max_steps**：防止死循环
- **正则解析**：提取 Thought/Action
- **异常处理**：工具调用失败

## 面试高频问题

**Q: 三种范式核心区别？**

> ReAct 动态决策灵活但易走偏；Plan-and-Solve 稳定但缺乏调整；Reflection 用额外调用换准确率。

**Q: 什么时候用 ReAct？**

> 需要外部工具、实时信息的任务。LLM 知识库没有实时数据，必须靠工具。

**Q: Agent 陷入循环怎么办？**

> 1. 加 max_steps 限制；2. 记录历史避免重复调用；3. 用 Reflection 做兜底检查。

**Q: 工具描述怎么写有效？**

> "当你需要查询实时信息时使用此工具"，而不是 "调用 SerpApi 传入 query 参数"。
