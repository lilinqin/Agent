# 智能体推理范式

> 适用场景：技术面试 / 技术架构设计 / 技术选型

---

## 核心概念（一句话定义）

推理范式是 Agent 在"思考→行动→反馈"循环中组织决策流程的方式——**范式决定 Agent 如何将长任务分解成 LLM 可操作的单步决策序列**。

---

## 三种范式对比

| 范式 | 思维模型 | 核心优势 | 核心代价 | 最适场景 |
|------|---------|----------|----------|----------|
| **ReAct** | 侦探式：走一步看一步 | 动态适应、可纠错 | 每步调 LLM，成本高 | 开放式任务、需外部工具 |
| **Plan-and-Solve** | 建筑师式：先画蓝图再施工 | 结构清晰、目标一致 | 规划错则全错，难以中途调整 | 逻辑确定的多步推理 |
| **Reflection** | 编辑式：写完初稿再校对 | 提升最终质量 | 调用翻倍，不适合实时场景 | 不容出错的高质量输出 |

---

## ReAct

### 核心机制

在每个时间步 $t$，策略（即 LLM）根据初始问题 $q$ 和历史轨迹生成思考 $th_t$ 和行动 $a_t$：

$$(th_t, a_t) = \pi(q, (a_1,o_1), \ldots, (a_{t-1},o_{t-1}))$$

环境工具执行行动并返回观察 $o_t = T(a_t)$，循环直到任务完成。

精髓在于：**LLM 负责"运筹帷幄"（推理），工具负责"解决具体问题"（执行）——Observation 是动态纠错的关键。**

### 三大优势

1. **高可解释性**：Thought 链完整记录决策过程，可追溯可调试
2. **动态纠错**：每步根据 Observation 调整，上一步搜索结果不好，立刻换词重搜
3. **工具协同**：天然支持多工具切换，突破 LLM 知识时效性和计算准确性限制

### 四大局限

1. **LLM 能力强依赖**：底层模型推理或遵循指令能力不足，流程就会中断
2. **执行效率低**：串行"思考→行动"，多步任务成本高、延迟大
3. **提示词脆弱**：整个机制建立在精心设计的模板上，措辞变化可能影响行为稳定性；正则解析也存在失败风险
4. **局部最优陷阱**：步进式决策缺少全局视野，可能在次优路径上越走越远甚至原地打转

### 适用场景

- 实时信息查询（天气、新闻、股价）
- 调用外部 API（查数据库、第三方服务）
- 需要根据中间结果动态调整的探索性任务

## Plan-and-Solve

### 核心机制

两阶段解耦——**先谋后动**：

**规划阶段**：LLM 将问题分解为 $n$ 个步骤的完整计划：
$$P = (p_1, p_2, \ldots, p_n) = \pi_{\text{plan}}(q)$$

**执行阶段**：按计划逐步完成，每步依赖前序结果：
$$s_i = \pi_{\text{solve}}(q, P, (s_1, \ldots, s_{i-1}))$$

精髓在于：**"先谋后动"让 Agent 在整个执行过程中保持更高的目标一致性，不会在中间步骤迷失方向。** 当代很多主流 Agent（如 Cursor、Claude Code）都融入了这种设计思想。

### 适用场景

- 多步数学应用题（先列解题步骤再逐一求解）
- 代码生成（先设计函数/类结构再逐一实现）
- 报告撰写（先列大纲再填充内容）

### 局限

- **规划错则全错**：初始计划质量决定最终成败，一行糟糕的计划会长出几百行糟糕的代码
- **缺乏动态调整**：执行中途发现问题很难回头修改；现实中通常需要配合"动态重规划"机制

## Reflection

### 核心机制

三步迭代循环：**执行 → 反思 → 优化**

反思模型对输出 $O_i$ 生成反馈 $F_i$：
$$F_i = \pi_{\text{reflect}}(\text{Task}, O_i)$$

优化模型据此生成更好的版本：
$$O_{i+1} = \pi_{\text{refine}}(\text{Task}, O_i, F_i)$$

循环直到"反馈无需改进"或达到最大迭代次数。

精髓在于：**为 Agent 引入"内部纠错回路"——不依赖外部工具反馈，而是自我审视逻辑层面的错误。** 迭代轨迹本身也形成"短期记忆"，记录从有缺陷的初稿到最终版本的完整演化路径。

### 成本收益分析

| 维度 | 说明 |
|------|------|
| **成本** | 每轮迭代至少额外 2 次 LLM 调用；任务延迟显著增加；提示工程复杂度上升 |
| **收益** | 方案质量从"合格"跃迁到"优秀"；修复逻辑漏洞和事实错误；大幅提升可靠性 |
| **适用** | 对质量、准确性有极高要求，实时性要求宽松（关键代码、技术报告、科研推演） |
| **不适用** | 需要快速响应，或"大致正确"已足够的场景 |

## 工程实践

### 范式选择决策树

```
任务是否需要外部工具或实时信息？
├── 是 → ReAct
└── 否 → 逻辑路径是否确定？
    ├── 确定 → Plan-and-Solve
    └── 不确定 → 对质量要求极高？
        ├── 是 → Reflection（叠在 ReAct/PaS 之上）
        └── 否 → ReAct（更轻量）
```

### 调试技巧（ReAct 特有）

- **打印完整提示词**：在每次 LLM 调用前打印完整上下文，追溯决策源头
- **分析原始输出**：解析失败时打印 LLM 原始文本，区分"LLM 没遵循格式"还是"解析逻辑有误"
- **验证工具 I/O**：确认 tool_input 格式和 observation 格式都符合预期
- **Few-shot 引导**：频繁出错时在提示词中加入完整的 TAO 成功案例

### 安全措施

- **max_steps**：防止死循环（必须有）
- **异常处理**：工具调用失败要有明确返回，不能吞掉错误
- **观察信号明确**：工具返回的 observation 必须区分"真的没找到"和"执行出错"

## 当前发展现状

**技术成熟度**：
- ReAct 是最成熟的范式，2022 年论文引用量极高
- Plan-and-Solve 在数学、代码场景效果好，已被主流工具融入
- Reflection 在"不容有错"场景是事实标准

**主要矛盾**：
- **成本 vs 效果**：Reflection 更准确但成本翻倍；ReAct 灵活但调用次数多
- **灵活 vs 稳定**：ReAct 灵活但不可预测，Plan-and-Solve 稳定但不灵活
- **规划质量瓶颈**：Harness Engineering 视角下，杠杆在规划环节，人应介入规划而非事后审核

**行业趋势**：
- 多范式组合成为主流：Plan + ReAct（执行阶段用 ReAct）+ Reflection（验证阶段）
- Boris Cherny（Claude Code 负责人）实践：任何非简单任务（3步以上）先进 plan mode，执行阶段清空 context 重新开始
- 研究→规划→执行→验证的四阶段分 session 运行成为顶级工程师的标配

---

## 未来展望

**短期（1-2年）**：
- 范式组合标准化：各阶段独立 session，互不污染 context
- 模型自身能力提升会内化部分 harness 逻辑（planning、self-verification）
- 评测标准化（GAIA、SWE-bench 等）

**长期（3-5年）**：
- "自适应推理"可能取代固定范式选择——模型根据任务自主决定用哪种范式
- Agentic RL 训练让模型在环境中自主学习哪种策略最优
- Multi-agent 场景下会出现跨 agent 的协同推理范式

## 深度理解

**Q: 三种范式的本质区别是什么？**

> 三者的分歧在于"何时决策"：ReAct 是实时决策（每步根据 Observation 决定下一步）；Plan-and-Solve 是预先决策（一次性规划好全部步骤）；Reflection 是事后决策（执行后回头审视并修正）。这对应三种不同的不确定性处理策略——当路径不确定时用 ReAct，当路径确定时用 PaS，当质量要求高时用 Reflection。

**Q: 为什么 ReAct 会陷入"局部最优"？**

> ReAct 是步进式贪心决策：每一步都在当前 Observation 基础上做局部最优，没有对整个任务的全局规划。这意味着一个不好的中间结果可能让 Agent 走错路，然后在这条错路上越陷越深——错误的观察导致错误的后续行动，累积偏差越来越大。这是所有贪心策略的固有缺陷。

**Q: 为什么 Plan-and-Solve 的"规划阶段"比执行阶段更重要？**

> Harness Engineering 的核心观察：一行糟糕的计划会生出几百行糟糕的代码，杠杆在上游。规划质量决定整个任务的天花板，而执行阶段只是在既定框架内填充。这也是为什么人工介入应该集中在规划阶段而非代码 review 阶段——前者的单位时间影响远高于后者。

## 一句话总结

ReAct 走一步看一步应对不确定性，Plan-and-Solve 先谋后动保证目标一致性，Reflection 事后迭代换取输出质量——**三种范式不互斥，实际系统通过组合覆盖"探索→规划→执行→验证"的完整决策链路。**
