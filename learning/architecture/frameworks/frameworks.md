# 框架对比

> 适用场景：技术面试、框架选型、系统设计

---

## 为什么要用框架

手写 Agent 代码（ReAct/Plan-and-Solve/Reflection）能帮你理解原理，但做产品会遇到瓶颈：

| 手写 | 框架 |
|------|------|
| 重复造轮子（循环、日志、状态管理） | 通用组件复用 |
| 多 Agent 协作难扩展 | 内置多 Agent 协调机制 |
| 调试靠 print | 事件回调可观测性 |

**框架本质**：提供经过验证的"规范"，把共性工作（Agent Loop、状态管理、工具调用、日志）抽象封装。

---

## 主流框架对比

| 框架 | 核心思路 | 适合场景 |
|------|----------|----------|
| **AutoGen** | 对话驱动协作 | 多角色软件团队、自动化工作流 |
| **AgentScope** | 消息驱动 + 工程化 | 大规模生产系统、分布式部署 |
| **CAMEL** | 角色扮演 + 引导提示 | 双 Agent 自主协作 |
| **LangGraph** | 图结构建模流程 | 复杂工作流、条件分支 |

---

## AutoGen：对话驱动

**核心理念**：把任务解决映射为不同角色 Agent 之间的自动对话。

### 核心组件

- **AssistantAgent**：任务解决者，封装 LLM，通过 System Message 赋予角色
- **UserProxyAgent**：人类代理，负责执行代码、调用工具、发出 TERMINATE 指令
- **RoundRobinGroupChat**：轮询群聊，按顺序让 Agent 发言（产品经理→工程师→审查员→测试）

### 面试话术

> "AutoGen 的优势是把复杂协作自然化为对话。不需要自己写状态机，定义好角色和发言顺序，框架自动完成流程控制。0.7.4 版本转向组合式架构 + 异步优先，并发处理能力更强。"

### 局限

- 对话本质有随机性，可能偏离预期分支
- 调试困难——没有堆栈，只有一长串对话记录

---

## AgentScope：消息驱动 + 工程化

阿里巴巴开源，专为大规模、高可靠性多智能体系统设计。

### 核心机制

- **消息驱动架构**：所有交互抽象为消息发送/接收，发送方和接收方时间解耦，天然支持高并发
- **MsgHub（消息中心）**：路由、持久化（SQLite/MongoDB）、分布式通信（RPC 透明处理）
- **生命周期管理**：基于 `AgentBase` 基类，开发者只需关注 `reply` 方法

### 面试话术

> "AgentScope 和 AutoGen 的根本区别是架构思路。AutoGen 是'对话工作室'，AgentScope 是'智能体操作系统'——它原生支持分布式部署、容错恢复、可观测性，适合需要长期稳定运行的生产环境。代价是学习曲线更陡，异步编程门槛更高。"

### 适用场景

- 需要高并发处理的多 Agent 交互
- 分布式部署的生产系统
- 需要持久化消息记录的可审计场景

---

## CAMEL：角色扮演 + 引导提示

**核心理念**：两个 Agent 通过"初始提示"（Inception Prompting）自主协作，不需要人类持续干预。

### 工作方式

1. **角色设定**：AI User（提需求） + AI Assistant（执行）
2. **引导提示**：在 System Prompt 中植入"行动纲领"——明确自身角色、告知协作者角色、定义共同目标、设定行为约束
3. **自主推进**：双方按约束规则对话，自动完成复杂任务

### 面试话术

> "CAMEL 解决了'如何让两个 AI 自主协作而不跑偏'的问题。关键是用精心设计的初始提示注入行为约束——要求 AI User 一次只提一个具体步骤，AI Assistant 完成上一步再问下一步，加上特定标记（如 `<SOLUTION>`）标识完成。这样即使没有人盯着，两个 Agent 也能高效往目标推进。"

---

## LangGraph：图结构

LangChain 生态成员，将 Agent 执行流程建模为**有向图**：

- **Node**：LLM 调用、工具执行等操作
- **Edge**：节点间的跳转逻辑

**核心优势**：天然支持**循环**，实现 Reflection 等迭代工作流特别直观。

## 自建框架

适用于教学、深度定制、特定领域优化。

**核心设计理念**：
- **轻量级**：极简依赖，可读性强
- **标准 API**：基于 OpenAI 接口标准
- **万物皆工具**：统一抽象，降低学习成本

---

## 选型决策

| 需求 | 推荐 |
|------|------|
| 快速原型、多 Agent 对话 | AutoGen |
| 生产级、分布式、高可靠 | AgentScope |
| 双 Agent 简单协作 | CAMEL |
| 复杂条件分支、循环工作流 | LangGraph |
| 零代码快速搭建 | Coze/Dify/n8n |

**混合方案最常见**：LangGraph 控主干流程保证确定性，在需要动态判断的节点嵌入 Agent 提供灵活性。

## 面试高频问题

**Q: 什么时候用手写 Agent 而不是框架？**
> "学习原理、或任务极其简单（只调一次 LLM）时手写可以。复杂任务、多 Agent 协作、生产环境必须用框架。"

**Q: AutoGen 和 AgentScope 的核心区别？**
> "AutoGen 是对话驱动，适合流程相对固定的多角色协作；AgentScope 是消息驱动 + 工程化，适合大规模分布式部署。"

**Q: 如何保证多 Agent 协作不跑偏？**
> "1. System Message 精确定义角色和行为约束；2. termination_condition 设置退出条件；3. max_turns 防止无限循环；4. 关键节点加入人工确认（Human-in-the-loop）。"

---

## 一句话总结

框架是把"如何运行 Agent"的共性逻辑抽象成可复用组件，让开发者专注"用什么 Agent、做什么任务"。选型本质是权衡开发效率和学习成本——简单任务用 AutoGen 快速上手，生产级系统用 AgentScope 的工程化能力，复杂流程用 LangGraph 的图模型表达能力。
