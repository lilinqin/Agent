# 框架对比

> 适用场景：技术面试 / 技术选型 / 架构设计

---

## 为什么要用框架

手写能帮助理解原理，但做产品会遇到瓶颈：

| 手写 | 框架 |
|------|------|
| 重复造轮子 | 通用组件复用 |
| 多 Agent 协作难扩展 | 内置协调机制 |
| 调试靠 print | 事件回调可观测性 |

框架的本质是提供经过验证的"规范"，把共性工作抽象封装。

---

## 主流框架对比

| 框架 | 核心思路 | 适合场景 |
|------|----------|----------|
| **AutoGen** | 对话驱动协作 | 多角色软件团队 |
| **AgentScope** | 消息驱动 + 工程化 | 大规模生产系统 |
| **CAMEL** | 角色扮演 + 引导提示 | 双 Agent 自主协作 |
| **LangGraph** | 图结构建模流程 | 复杂工作流 |

---

## AutoGen

核心理念是：**把任务解决映射为不同角色 Agent 之间的自动对话。**

- AssistantAgent：任务解决者
- UserProxyAgent：人类代理
- RoundRobinGroupChat：轮询群聊

**优势**：把复杂协作自然化为对话，不需要自己写状态机。0.7.4 版本转向组合式架构 + 异步优先。

**局限**：对话有随机性，可能偏离预期分支；调试困难。

---

## AgentScope

阿里巴巴开源，专为**大规模、高可靠性**多智能体系统设计。

- 消息驱动架构：发送方和接收方时间解耦，天然支持高并发
- MsgHub：路由、持久化、分布式通信
- 原生支持分布式部署、容错恢复

**判断**：AutoGen 是"对话工作室"，AgentScope 是"智能体操作系统"。

---

## LangGraph

将 Agent 执行流程建模为**有向图**：
- Node：LLM 调用、工具执行
- Edge：节点跳转逻辑

核心优势：**天然支持循环**，实现 Reflection 等迭代工作流特别直观。

---

## 选型决策

| 需求 | 推荐 |
|------|------|
| 快速原型 | AutoGen |
| 生产级分布式 | AgentScope |
| 复杂工作流 | LangGraph |
| 双 Agent 协作 | CAMEL |

我们目前的实践：**混合方案最常见**——LangGraph 控主干，动态节点嵌入 Agent。

---

## 当前发展现状

**技术成熟度**：
- 框架生态百花齐放
- 但都还在快速迭代中
- 没有绝对王者

**主要矛盾**：
- **灵活性 vs 易用性**：功能强的框架往往学习曲线高
- **稳定 vs 迭代**：API 变化快，升级成本高
- **定制 vs 通用**：过度定制难以迁移

**行业趋势**：
- 框架趋于收敛（学 LangGraph 的图模型）
- 多 Agent 协作成为热点
- 评测体系开始建立

---

## 未来展望

**短期（1-2年）**：
- 框架会趋于稳定
- 多 Agent 协作会更成熟
- 会出现评测标准

**长期（3-5年）**：
- 框架可能"消失"（被 LLM 平台吸收）
- MCP 等协议统一底层
- Agent 开发像现在写 SQL 一样简单

**我们的布局**：
- 核心：LangGraph 为基础
- 策略：保持框架独立性，通过 MCP 兼容
- 储备：关注新型框架发展

## 面试常见问题

**Q: 什么时候用手写 Agent 而不是框架？**

> 学习原理、或任务极其简单（只调一次 LLM）时手写可以。复杂任务、多 Agent 协作、生产环境必须用框架。

**Q: AutoGen 和 AgentScope 的核心区别？**

> AutoGen 是对话驱动，适合流程相对固定的多角色协作；AgentScope 是消息驱动 + 工程化，适合大规模分布式部署。

**Q: 如何保证多 Agent 协作不跑偏？**

> 1. System Message 精确定义角色和行为约束；2. termination_condition 设置退出条件；3. max_turns 防止无限循环；4. 关键节点人工确认。

## 一句话总结

框架是把"如何运行 Agent"的共性逻辑抽象成可复用组件，让开发者专注"用什么 Agent、做什么任务"。选型本质是权衡开发效率和学习成本。
