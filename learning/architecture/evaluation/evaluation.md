# 智能体评估体系

> 适用场景：技术面试 / 系统质量保障 / 模型选型 / 上线决策

---

## 核心概念（一句话定义）

Agent 评估是用标准化方法量化 Agent 能力的工程实践——**没有评估，就没有可信的改进；有了评估，才能把"感觉变好了"转化为"确实提升了 X%"**。

---

## 为什么 Agent 评估困难

Agent 评估比传统软件测试面临更多挑战：

1. **输出不确定性**：同一问题可能有多个正确答案，难以用简单对错判断
2. **评估标准多样**：工具调用需要检查函数签名；问答需要评估语义相似度；长程任务需要评估中间步骤
3. **评估成本高**：每次评估需要大量 API 调用，完整跑一套基准可能花费数百元
4. **过拟合风险**：针对特定 benchmark 优化可能导致实际场景表现下降

## 主流评估基准

### 工具调用能力

| 基准 | 来源 | 规模 | 特点 |
|------|------|------|------|
| **BFCL** (Berkeley Function Calling Leaderboard) | UC Berkeley | 1120+ 样本 | 覆盖 simple/multiple/parallel/irrelevance 四类；AST 匹配算法；社区活跃 |
| **ToolBench** | 清华大学 | 16000+ 场景 | 真实 API 调用，覆盖复杂工具使用 |
| **API-Bank** | Microsoft Research | 53 个 API | 专注评估对 API 文档的理解能力 |

BFCL 四类的含义：
- **simple**：单个工具调用，参数简单
- **multiple**：多个工具可选，需要选对一个
- **parallel**：需要并行调用多个工具
- **irrelevance**：工具与问题无关，应拒绝调用

### 通用能力评估

| 基准 | 来源 | 特点 |
|------|------|------|
| **GAIA** (General AI Assistants) | Meta AI + Hugging Face | 466 道真实问题，Level 1-3 难度分级，评估多步推理/工具使用/文件处理/网页浏览 |
| **AgentBench** | 清华大学 | 8 个领域任务，评估通用 Agent 能力 |
| **SWE-bench** | Princeton | 评估代码 Agent 解决 GitHub Issue 的能力 |
| **WebArena** | CMU | 评估 Agent 在真实网页环境中的任务完成能力 |

### 多智能体协作评估

- **ChatEval**：评估多 Agent 对话系统质量
- **SOTOPIA**：评估 Agent 在社交场景中的互动能力

## 常用评估指标

### 准确性指标

| 指标 | 含义 | 适用场景 |
|------|------|---------|
| Accuracy | 正确率 | 有明确答案的任务 |
| Exact Match | 精确匹配 | 工具调用参数验证 |
| F1 Score | 精确率与召回率的调和平均 | 信息抽取、文本生成 |
| Quasi Exact Match | 允许等价表达式 | GAIA 等真实场景 |

### 效率指标

- **Response Time**：端到端响应延迟
- **Token Usage**：单任务 token 消耗，影响成本
- **Step Count**：完成任务所需的 Agent 循环次数

### 鲁棒性指标

- **Error Rate**：工具调用失败率
- **Failure Recovery**：遇到错误后的恢复能力
- **Consistency**：相同输入的输出一致性

## 评估方式

### LLM Judge

用一个强大的 LLM（如 GPT-4）来评判另一个 Agent 的输出质量。

优点：能评估开放性问题，不需要标准答案
缺点：有偏见（同 family 模型互评可能虚高），成本高

### Win Rate

让两个 Agent 在相同任务上 PK，人工或 LLM 判断哪个更好。适合模型版本间比较。

### 行为轨迹分析

不只看最终结果，分析 Agent 的决策路径：
- 工具调用序列是否合理？
- 有没有无效循环？
- 资源（token）利用效率如何？

## 工程实践

### 先建评估，再做优化

> "先记录，后优化"——每次 Agent 出问题，先看 Trace 找根因，再修改

常见反模式：Agent 表现变差，立刻改提示词。但如果不看 Trace，你不知道是工具描述不清、上下文太长还是模型本身的问题——改错了方向，越改越差。

### 单变量改动

每次只改一个因素（提示词/工具描述/记忆策略），然后用相同的评估集对比前后效果。一次改多个因素，你永远不知道是哪个起了作用。

### 黄金测试集

维护一个覆盖核心场景的"黄金测试集"：
- 包含成功案例和边界情况
- 每次改动后必须跑全量测试
- 新增 Bug 修复后，将这个 case 加入测试集（防止回归）

这正是 Harness Engineering 中"反馈闭环"的工程实现。

### 评估 vs 过拟合

警惕：针对 benchmark 优化和针对真实场景优化可能是两件事。GAIA Level 3 能拿高分不代表生产环境表现好。建议同时维护：
- 公开 benchmark（对齐行业水位）
- 内部真实业务测试集（对齐实际效果）

## 当前发展现状

**技术成熟度**：
- 工具调用评估（BFCL）最成熟，有明确排行榜
- 通用 Agent 评估（GAIA、SWE-bench）快速成熟
- 多 Agent 协作评估仍在早期

**主要矛盾**：
- **评估成本 vs 评估覆盖**：跑完整基准代价高，只跑子集可能漏掉边界情况
- **静态基准 vs 动态能力**：Agent 能力在快速提升，固定基准很快过时
- **自动评估 vs 人工判断**：LLM Judge 可扩展但有偏见，人工精准但成本高

**行业趋势**：
- 评测成为模型公司的核心竞争力（模型选型依赖 leaderboard）
- 企业开始建立自己的领域评测集（公开基准≠业务表现）
- Harness 轨迹数据正在成为评测的新素材

## 未来展望

**短期（1-2年）**：
- Agentic 评测基准快速扩展（从单步到多步、从单 Agent 到多 Agent）
- 评测即数据：评测集的质量成为模型迭代的关键
- 行为轨迹评估（不只看结果，也看路径）

**长期（3-5年）**：
- 自动化的"评估智能体"出现，可动态生成测试用例
- 评测和训练深度耦合（Agentic RL：用评测结果直接驱动训练）

## 深度理解

**Q: 为什么 Agent 评估比传统软件测试难？**

> 传统测试有确定性的输入输出，对错一目了然。Agent 的难点在于：输出的多样性（同一问题多个正确答案）、路径的不确定性（达到目标的方式千变万化）、以及能力的涌现性（好的 Agent 可能用了你完全没想到的工具组合）。这决定了 Agent 评估必须同时评估"结果"和"过程"。

**Q: BFCL 和 GAIA 评的是同一件事吗？**

> 不是。BFCL 评的是"精准调用"——给定场景，能否正确选工具、构造参数；GAIA 评的是"综合能力"——面对真实问题，能否通过多步推理、工具组合、信息整合最终得出正确答案。BFCL 是局部能力测试，GAIA 是综合任务测试。实际系统需要两者兼顾。

## 一句话总结

Agent 评估的本质是把"感觉"变成"数据"——**先有黄金测试集才能判断改动是进步还是退步；先有行为轨迹分析才能找到 Agent 失败的真正原因，而不是靠提示词"玄学调参"**。
